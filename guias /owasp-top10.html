// Fundamentos

¿Qué es el OWASP Top 10?

El OWASP Top 10 es el documento de concienciación estándar sobre seguridad en aplicaciones web. Representa el consenso sobre los riesgos más críticos y es el punto de partida obligatorio para cualquier programa de seguridad de aplicaciones.

// NO ES UNA CHECKLIST COMPLETA
El Top 10 es un punto de partida, no un estándar exhaustivo. Para cobertura completa, combínalo con OWASP ASVS (Application Security Verification Standard) que define más de 286 controles en 3 niveles. El Top 10 cubre lo que el ASVS Nivel 1 detecta automáticamente.
// 3 CAMBIOS CLAVE EN 2021
A04 Diseño Inseguro — nuevo: proactivo, no solo reactivo (Threat Modeling desde el inicio).
A08 Integridad de Software y Datos — nuevo: supply chain, CI/CD, deserialización.
A10 SSRF — nuevo: ascendió por votación directa de la comunidad.
XSS fue fusionado dentro de A03 Inyección.
// Por qué importa el Top 10
  1. El 90% de los incidentes web explotó al menos una de las 10 categorías
  2. Referencia obligatoria en PCI DSS, ISO 27001 y auditorías de seguridad
  3. Base de herramientas SAST/DAST: SonarQube, Burp Suite, OWASP ZAP
  4. Lenguaje común entre devs, QA y equipos de seguridad
  5. Requisito en la mayoría de RFPs de desarrollo de software seguro

01
A01:2021
Pérdida de Control de Acceso
CWE-200CWE-352
▼
Las restricciones sobre qué pueden hacer los usuarios autenticados no se aplican correctamente. Subió al #1 en 2021 con el 94% de aplicaciones teniendo alguna forma de control de acceso roto. Los atacantes explotan estos fallos para acceder a cuentas ajenas, archivos sensibles o funciones administrativas.
// VECTORES
  • IDOR: cambiar IDs en URL para ver datos de otros usuarios
  • Forzar navegación a URLs de administración sin rol admin
  • CSRF: ejecutar acciones autenticadas desde otro sitio
  • Elevar privilegios actuando como otro usuario sin login
// VULNERABLE
// URL del usuario legítimo
GET /api/facturas/1001

// Atacante cambia el ID
GET /api/facturas/1002
// → accede a datos ajenos
// sin verificar propiedad
// DEFENSAS
  • Denegar por defecto todo acceso no permitido explícitamente
  • Verificar propiedad del recurso en cada request
  • RBAC o ABAC con modelo de permisos centralizado
  • Tokens CSRF en operaciones de cambio de estado
  • Auditar y testear controles de acceso regularmente
02
A02:2021
Fallos Criptográficos
CWE-259CWE-327
▼
Fallos criptográficos que llevan a exposición de datos sensibles. Antes llamada "Exposición de Datos Sensibles". Incluye transmisión sin cifrar, algoritmos débiles, contraseñas mal almacenadas y secretos hardcodeados en el código fuente.
// VECTORES
  • Contraseñas con MD5/SHA1 sin salt → rainbow tables
  • Claves API en repositorios públicos de GitHub
  • Datos sensibles en HTTP sin TLS
  • SSL/TLS obsoleto (SSLv3, TLS 1.0) o cipher débiles
// VULNERABLE
// ❌ Hash sin salt
$h = md5($password);

// ❌ Secreto hardcodeado
$key = "sk-prod-abc123xyz";

// ✅ Correcto
$h = password_hash($pwd,
  PASSWORD_ARGON2ID);
$key = $_ENV['API_KEY'];
// DEFENSAS
  • bcrypt, Argon2id o scrypt para contraseñas — nunca MD5
  • TLS 1.3 + HSTS; deshabilitar versiones antiguas
  • Secretos en variables de entorno o vault (HashiCorp, AWS SSM)
  • AES-256-GCM para datos sensibles en reposo
  • Secret scanning en CI/CD (GitLeaks, TruffleHog)
03
A03:2021
Inyección (incl. XSS)
CWE-79CWE-89CWE-73
▼
Datos del usuario se envían a un intérprete como parte de un comando o consulta. Incluye SQL, XSS, Command, LDAP e XPATH Injection. El 94% de aplicaciones fue testado. XSS fue unificado en esta categoría en 2021.
// VECTORES
  • SQL Injection: extraer, modificar o eliminar datos de BD
  • XSS Stored/Reflected: robo de sesiones, defacement
  • Command Injection: ejecutar comandos del SO
  • LDAP Injection: bypass de autenticación en directorios
// SQL INJECTION
// ❌ Concatenación directa
$q = "SELECT * FROM users
WHERE user='".$_GET['u']
."' AND pass='".$_GET['p']."'";
// Payload: ' OR '1'='1
// → bypass de autenticación

// ✅ Parametrizado
$stmt=$pdo->prepare(
"SELECT * FROM users
WHERE user=? AND pass=?");
$stmt->execute([$u,$p]);
// DEFENSAS
  • Prepared statements para toda interacción con BD
  • ORM con escapado automático (Eloquent, Hibernate)
  • Validar y sanitizar TODOS los inputs en servidor
  • Para XSS: escapar output según contexto (HTML, JS, URL)
  • Content Security Policy (CSP) en cabeceras HTTP
04
A04:2021 🆕
Diseño Inseguro
CWE-209CWE-501
▼
Riesgos de diseño y arquitectura. Un diseño inseguro no se puede arreglar con una implementación perfecta. Se enfoca en la falta de threat modeling, ausencia de patrones de diseño seguro y casos de abuso no considerados desde el principio.
// FALLOS DE DISEÑO
  • Recuperación de contraseña que revela si el email existe
  • Sin límite de intentos en funciones críticas (fuerza bruta)
  • Lógica de negocio con estados inválidos explotables
  • Mensajes de error con información interna del sistema
// ENUMERACIÓN DE USUARIOS
// ❌ Revela si existe el email
POST /forgot-password
→ "Email no registrado"
→ "Revisa tu correo"
// permite enumerar usuarios

// ✅ Respuesta genérica siempre
→ "Si el email existe,
recibirás un enlace"
// misma respuesta siempre
// DEFENSAS DE DISEÑO
  • Threat Modeling (STRIDE) en fase de diseño, antes de codificar
  • Mensajes de error genéricos que no revelan el sistema
  • Rate limiting en el diseño de todas las funciones sensibles
  • Casos de abuso en los requisitos de negocio
  • Principio de mínimo privilegio en la arquitectura
05
A05:2021
Configuración de Seguridad Incorrecta
CWE-16CWE-611
▼
La categoría más prevalente: el 90% de las aplicaciones fue testeada. Incluye configuraciones por defecto inseguras, características innecesarias habilitadas, stack traces expuestos, headers de seguridad faltantes y permisos incorrectos en la nube.
// CONFIGURACIONES PELIGROSAS
  • Panel admin con credenciales por defecto (admin/admin)
  • Bucket S3 con lectura pública a datos de clientes
  • Stack traces de PHP/Java en producción
  • Headers de seguridad ausentes (HSTS, CSP, X-Frame)
  • Directory listing habilitado en el servidor web
// HEADERS SEGUROS
// ✅ Headers obligatorios
Strict-Transport-Security:
  max-age=31536000;
  includeSubDomains
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Content-Security-Policy:
  default-src 'self'
Permissions-Policy:
  geolocation=()
Referrer-Policy:
  strict-origin
// DEFENSAS
  • Hardening automatizado con IaC (Ansible, Terraform)
  • Verificar securityheaders.com periódicamente
  • Deshabilitar features y módulos no usados
  • Revisar permisos cloud: ScoutSuite, Prowler
  • Errores genéricos en producción; detalles solo en logs internos
06
A06:2021
Componentes Vulnerables y Desactualizados
CWE-1104
▼
Librerías, frameworks y componentes con los mismos privilegios que la aplicación. Si un componente tiene una CVE, toda la aplicación está en riesgo. Log4Shell (CVE-2021-44228) afectó a millones de sistemas con una RCE trivial en una librería de logging.
// CASOS REALES
  • Log4Shell: RCE en Log4j usado por miles de apps Java
  • Heartbleed: lectura de memoria del servidor via OpenSSL
  • Struts2 RCE: breach de Equifax, 147M personas afectadas
  • NPM packages maliciosos: event-stream, ua-parser-js
// AUDITORÍA DE DEPS
# Auditar vulnerabilidades
npm audit
pip-audit
mvn dependency-check:check

# SBOM (inventario)
syft . -o spdx-json > sbom.json
grype sbom:sbom.json

# GitHub Dependabot
# (configurar en .github/
#  dependabot.yml)
// DEFENSAS
  • SBOM: inventario de todos los componentes con versiones
  • Monitoring automático de CVEs (Dependabot, Snyk)
  • Política de parches: crítico <7 días, alto <30 días
  • Verificar integridad de paquetes (checksums, firma GPG)
  • Preferir componentes con historial activo de seguridad
07
A07:2021
Fallos de Identificación y Autenticación
CWE-287CWE-384
▼
Fallos en la confirmación de identidad, autenticación y gestión de sesiones. Permite a los atacantes comprometer contraseñas, tokens o asumir la identidad de otros usuarios temporal o permanentemente.
// VECTORES
  • Credential stuffing con listas de brechas anteriores
  • Fuerza bruta sin rate limiting ni bloqueo progresivo
  • Contraseñas por defecto no cambiadas (admin/admin)
  • Session tokens en URL o con entropía insuficiente
  • Session fixation: servidor acepta IDs externos
// SESIÓN INSEGURA
// ❌ Session ID en URL
GET /dashboard?sid=abc123

// ❌ Sin regenerar tras login
session_start();
$_SESSION['user']=$user;
// falta: session_regenerate_id()

// ✅ Correcto
session_start();
session_regenerate_id(true);
$_SESSION['user']=$user;
// DEFENSAS
  • MFA obligatorio para cuentas privilegiadas y datos sensibles
  • Tokens de sesión: 128+ bits, HttpOnly, Secure, SameSite=Strict
  • Invalidar sesión en logout del lado servidor
  • Rate limiting y CAPTCHA en intentos de login fallidos
  • Verificar credenciales en HaveIBeenPwned al registrar

A08:2021 🆕
Fallos en Software e Integridad de Datos
CWE-829CWE-494
▼
Suposiciones sobre integridad de actualizaciones, datos críticos y pipelines CI/CD sin verificación. Incluye deserialización insegura y ataques a la cadena de suministro de software. SolarWinds es el caso más famoso: el build pipeline fue comprometido.
// VECTORES
  • Actualización de software sin verificar firma digital
  • Deserialización de objetos no confiables → RCE
  • CI/CD comprometido: código malicioso inyectado en builds
  • Typosquatting en paquetes NPM/PyPI/RubyGems
// DESERIALIZACIÓN
// ❌ Java: RCE posible
ObjectInputStream ois =
  new ObjectInputStream(
    request.getInputStream());
Object obj = ois.readObject();

// ❌ PHP
$data=unserialize($_COOKIE['u']);

// ✅ Usar JSON + schema
$d = json_decode($input);
$v->validate($d, $schema);
// DEFENSAS
  • Firmar y verificar paquetes de software antes de instalar
  • Evitar deserializar datos no confiables; preferir JSON
  • Asegurar el pipeline CI/CD: acceso mínimo, firmar commits
  • Subresource Integrity (SRI) para scripts de CDNs externos
  • SBOM para rastrear toda la cadena de suministro
09
A09:2021
Fallos en el Registro y Monitoreo de Seguridad
CWE-778CWE-117
▼
Sin registro y monitoreo adecuados, los ataques no se detectan. El tiempo promedio de detección de una brecha es 197 días. El 47% de las brechas se detectan por terceros, no por las organizaciones afectadas. Esta categoría apoya la detección, escalado y análisis forense.
// FALLOS COMUNES
  • Eventos auditables no registrados (logins, fallos, transacciones)
  • Logs sin contexto suficiente para análisis forense
  • Logs solo locales — se pierden si el atacante los borra
  • Sin alertas para actividad sospechosa (muchos 401s seguidos)
  • Logs con datos sensibles (contraseñas, tokens) — compliance fail
// LOG CORRECTO
// ✅ Estructura correcta
{
"ts":"2025-03-15T10:42Z",
"level":"WARN",
"event":"login_failed",
"user_id":"usr_456",
"ip":"203.0.113.10",
"attempts":5,
"req_id":"req_abc123"
// sin contraseña nunca
}
// DEFENSAS
  • Registrar: logins, fallos auth, errores validación, cambios de privilegios
  • Centralizar logs en SIEM (Elastic, Splunk) — no solo archivos locales
  • Alertas automáticas: N fallos login, IPs nuevas, acceso fuera de horario
  • Retención: 90 días activo, 1 año en archivo
  • Nunca loguear contraseñas, tokens ni datos de tarjetas

A10:2021 🆕
SSRF — Server-Side Request Forgery
CWE-918
▼
Ocurre cuando una aplicación obtiene un recurso remoto sin validar la URL del usuario. Permite hacer que el servidor realice requests hacia destinos arbitrarios — incluyendo servicios internos, metadata de cloud (AWS 169.254.169.254) y localhost. Entró al Top 10 directamente por votación de la comunidad.
// VECTORES
  • Acceder al metadata AWS para obtener credenciales IAM
  • Escanear la red interna desde el servidor (port scan)
  • Acceder a servicios internos no expuestos (Redis, Elastic)
  • Bypass de firewalls usando el servidor como proxy
// SSRF EN AWS
// Función legítima: preview URL
POST /api/preview
{"url":"https://example.com"}

// Ataque SSRF
POST /api/preview
{"url":"http://169.254.169.254/
latest/meta-data/iam/
security-credentials/role"}

// → Obtiene credenciales IAM
// → Acceso total a AWS
// DEFENSAS
  • Whitelist de dominios/IPs permitidos para requests externos
  • Bloquear IPs privadas RFC1918 y 169.254.169.254
  • No seguir redirecciones HTTP automáticamente
  • Segmentación de red: servidor web sin acceso a servicios internos
  • En AWS: usar IMDSv2 que requiere token de sesión

OWASP en el SDLC

Corregir una vulnerabilidad cuesta 100x más en producción que en diseño. El OWASP Top 10 debe estar integrado en cada fase, no solo en pruebas finales.

// DISEÑO & REQUISITOS
Threat Modeling con STRIDE para identificar A04 (Diseño Inseguro). Definir casos de abuso junto a los de uso. Requisitos de seguridad explícitos en el backlog.
// DESARROLLO
SAST en el IDE (SonarLint, ESLint security plugins). Code review con Top 10 como checklist. Formación en secure coding específica del lenguaje.
// CI/CD PIPELINE
SAST automático en cada commit (Semgrep, SonarQube). SCA para dependencias (Snyk, Dependabot). Secret scanning (GitLeaks). DAST en staging (OWASP ZAP).
// PRODUCCIÓN
WAF activo (A03, A04). SIEM con alertas (A09). Pentesting periódico. Bug Bounty. Gestión de vulnerabilidades con SLAs de remediación.
// TIPOS DE ANÁLISIS
SAST — análisis de código fuente estático. Detecta en desarrollo. Herramientas: Semgrep, SonarQube, Bandit (Python), Brakeman (Rails).

DAST — testeo de la app en ejecución, desde fuera. Herramientas: OWASP ZAP, Burp Suite, Nikto.

IAST — agentes dentro de la app, combina ambos. Herramientas: Contrast Security.

SCA — análisis de composición de software (dependencias). Herramientas: Snyk, OWASP Dependency-Check, Trivy.
// OWASP ASVS — EL SIGUIENTE NIVEL
El ASVS (Application Security Verification Standard) tiene 286 controles en 3 niveles:
L1: Pentesting básico → cubre Top 10 detectable automáticamente
L2: La mayoría de apps → para datos sensibles y transacciones
L3: Apps críticas → banca, salud, infraestructura

Relación con Otros Estándares

ESTÁNDARRELACIÓN CON OWASP TOP 10
ISO 27001:2022Los controles 8.25–8.29 del Anexo A (desarrollo seguro, testing, programación segura, SSDLC) se implementan aplicando el OWASP Top 10 como referencia técnica principal.
NIST CSF 2.0PR.PS-06 (código seguro) e ID.RA-01 (vulnerabilidades) del CSF se implementan con el Top 10 y ASVS. La función PROTECT del CSF mapea directamente a defensas contra el Top 10.
PCI DSS v4.0PCI DSS exige explícitamente testing contra OWASP Top 10 para apps que procesan datos de tarjetas (requerimientos 6.2 y 6.3). Es un requisito de cumplimiento directo.
OWASP ASVSEl ASVS es el nivel siguiente: 286 controles en 3 niveles. El Top 10 es el subconjunto mínimo que cualquier app debe cumplir; el ASVS es el estándar completo de verificación de seguridad de aplicaciones.
MITRE ATT&CKT1190 (Exploit Public-Facing App), T1059 (Command Injection) y T1539 (Steal Web Session Cookie) de ATT&CK mapean directamente a A03, A01 y A07 del OWASP Top 10.
Ley 21.663 ChileLos OIV que operan aplicaciones web deben implementar controles del OWASP Top 10 como evidencia de medidas preventivas según el Art. 7° y para cumplir los requisitos del SGSI (ISO 27001).
