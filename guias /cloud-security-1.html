// Conceptos Base

Fundamentos de Seguridad Cloud

La nube no es inherentemente menos segura que un datacenter propio — pero los riesgos son distintos y los controles deben adaptarse. El mayor problema no es que el proveedor cloud sea inseguro: es que los clientes configuran mal sus propios recursos.

// GARTNER: EL 99% DE LAS BRECHAS CLOUD
Hasta 2025, el 99% de los fallos de seguridad en la nube serán culpa del cliente, no del proveedor (Gartner). Las causas más frecuentes: buckets S3 públicos por error de configuración, permisos IAM excesivos, credenciales hardcodeadas en código, ausencia de MFA, y grupos de seguridad demasiado permisivos. La infraestructura del proveedor es segura — la configuración que hace el cliente no siempre lo es.
// LOS 3 MODELOS DE SERVICIO
IaaS — Infrastructure as a Service: EC2, Azure VMs, GCE. Más control, más responsabilidad del cliente.

PaaS — Platform as a Service: RDS, App Service, Cloud Run. Responsabilidad compartida en middleware y runtime.

SaaS — Software as a Service: M365, Salesforce, Slack. El proveedor gestiona todo excepto los datos y el acceso de usuarios.
// Por qué la seguridad cloud es diferente
  1. Los recursos se crean y destruyen en segundos — el inventario es dinámico, no estático
  2. Todo está potencialmente expuesto a Internet por defecto — superficie de ataque global
  3. Las APIs son la interfaz principal — cada endpoint mal protegido es una puerta de entrada
  4. Las identidades (IAM) son el nuevo perímetro — una clave comprometida da acceso amplio
  5. Los errores de configuración escalan masivamente — un terraform mal configurado despliega cientos de recursos vulnerables
  6. La visibilidad requiere configuración activa — los logs no están habilitados por defecto en todos los servicios
  7. Multi-cloud y entornos híbridos multiplican la complejidad de gestión
// El Concepto Más Importante

Modelo de Responsabilidad Compartida

El modelo de responsabilidad compartida define qué aspectos de seguridad gestiona el proveedor cloud y cuáles son responsabilidad del cliente. Muchas brechas ocurren porque el cliente asume que el proveedor lo protege de algo que en realidad es responsabilidad propia.

// LO QUE SIEMPRE ES RESPONSABILIDAD DEL CLIENTE
Sin importar el modelo (IaaS, PaaS o SaaS), el cliente siempre es responsable de: sus datos, la gestión de identidades y accesos (IAM), la configuración de seguridad de sus recursos, el cifrado de sus datos con sus propias claves, y el cumplimiento regulatorio aplicable a su sector.
// CSA Top Threats to Cloud Computing 2022

Las 12 Principales Amenazas Cloud

La Cloud Security Alliance (CSA) publica periódicamente el informe "Egregious Eleven / Pandemic Eleven" con las amenazas más críticas en entornos cloud. La versión 2022 identifica 12 amenazas principales.

AMENAZA #1
Infraestructura y APIs inseguras
APIs de gestión cloud mal aseguradas expuestas a Internet. Falta de autenticación fuerte, claves de API sin rotación, endpoints sin rate limiting.
→ Autenticación en todas las APIs · Rate limiting · Rotación automática de claves · WAF en APIs públicas
AMENAZA #2
Mala configuración y control inadecuado del cambio
El error humano en la configuración es la causa más frecuente de brechas cloud. Buckets S3 públicos, grupos de seguridad 0.0.0.0/0, bases de datos sin autenticación.
→ CSPM (Cloud Security Posture Management) · IaC seguro · Revisión de configuración automatizada
AMENAZA #3
Arquitectura cloud y estrategia de seguridad deficientes
Migrar "lift and shift" sin adaptar la arquitectura al modelo cloud. No aprovechar los controles nativos del proveedor. Sin estrategia de seguridad cloud explícita.
→ Well-Architected Framework · Diseño cloud-native · Security by design desde el principio
AMENAZA #4
Gestión insuficiente de identidades, credenciales y accesos
Claves de acceso de largo plazo, roles IAM con permisos excesivos (AdministratorAccess), sin MFA en cuentas root, credenciales hardcodeadas en código.
→ Principio de mínimo privilegio · MFA obligatorio · Roles en lugar de claves · Secretos en vault
AMENAZA #5
Secuestro de cuentas
Compromiso de credenciales cloud mediante phishing, credential stuffing o claves expuestas en repositorios GitHub. El atacante asume control total de la cuenta cloud.
→ MFA en todas las cuentas · Alertas de comportamiento anómalo · UEBA · Rotación de credenciales
AMENAZA #6
Amenazas internas
Empleados, contratistas o socios con acceso a recursos cloud que actúan maliciosamente o por descuido. El acceso desde cualquier lugar amplifica el riesgo.
→ Mínimo privilegio · Monitoreo de comportamiento · Just-in-time access · PAM · Revisión de accesos
AMENAZA #7
Interfaces inseguras de terceros
Integraciones con servicios de terceros, herramientas de automatización y APIs de partners que tienen acceso a recursos cloud con permisos excesivos o sin monitoreo.
→ Inventario de integraciones · Permisos mínimos para terceros · Revisión periódica de OAuth grants
AMENAZA #8
Plano de control débil
Acceso no protegido a la consola de gestión del proveedor cloud. Sin separación entre cuentas de producción y desarrollo. Acceso root sin restricciones.
→ Cuenta root solo para emergencias · Cuentas separadas por entorno · SCPs en AWS Organizations
AMENAZA #9
Fallos en metastructura y applistructura
Vulnerabilidades en los servicios propios del proveedor (metastructura) o en las capas de abstracción de las aplicaciones cloud-native que el cliente no puede parchear directamente.
→ Monitoreo de notificaciones del proveedor · Defense in depth · Arquitectura resiliente ante fallos de un servicio
AMENAZA #10
Visibilidad limitada del uso de la nube
Shadow IT cloud: empleados crean recursos cloud sin aprobación, fuera del inventario oficial y sin controles de seguridad. Datos sensibles en cuentas cloud no gestionadas.
→ CASB para descubrir shadow IT · Cloud governance · Inventario centralizado · Políticas de uso aceptable
AMENAZA #11
Abuso y uso malicioso de servicios cloud
Atacantes que usan la infraestructura cloud legítima para alojar malware, C2 (command and control), minería de criptomonedas o lanzar ataques DDoS.
→ Monitoreo de costos anómalos · Alertas de CloudTrail/Activity Log · Threat Intelligence feeds
AMENAZA #12
Riesgos de la cadena de suministro cloud
Dependencias de servicios cloud de terceros, imágenes de contenedores de orígenes no confiables, módulos Terraform públicos con código malicioso o vulnerable.
→ Verificar fuentes de imágenes · Escanear imágenes de contenedores · SBOM · Usar solo registros de confianza
// CIS Benchmarks — Controles por Proveedor

Controles Fundamentales CIS Cloud

El Center for Internet Security (CIS) publica Benchmarks específicos para cada proveedor cloud con configuraciones recomendadas. Son el estándar de hardening más adoptado para entornos cloud. Selecciona el proveedor para ver los controles prioritarios.

// CIS AWS FOUNDATIONS BENCHMARK v2.0
Organizado en 5 secciones: IAM, Almacenamiento, Logging, Monitoreo, Redes. Nivel 1 (controles esenciales sin impacto en funcionalidad) y Nivel 2 (controles adicionales para entornos más restrictivos).
1.1No usar la cuenta root para actividades cotidianas
La cuenta root tiene acceso irrevocable a todos los recursos. Crear usuarios IAM individuales. Guardar las credenciales root en un lugar seguro offline.
IAMNivel 1Crítico
1.5MFA habilitado en la cuenta root
La cuenta root debe tener MFA de hardware (YubiKey o similar). Sin MFA, el compromiso de las credenciales root da acceso total sin forma de recuperarlo.
IAMNivel 1Crítico
1.10MFA habilitado para todos los usuarios IAM con consola
Todo usuario IAM que acceda a la consola web debe tener MFA activo. Aplicar mediante política de IAM que deniegue el acceso si MFA no está configurado.
IAMNivel 1
1.14Eliminar claves de acceso de larga duración no usadas
Las claves de acceso de usuarios IAM que no se usen en 90 días deben desactivarse y eliminarse. Preferir roles IAM sobre claves de larga duración.
IAMNivel 1
1.17Asegurar que solo admins gestionan políticas de IAM
Evitar que usuarios no administradores puedan crear o modificar políticas IAM que les concedan más permisos (privilege escalation vía IAM).
IAMNivel 1
2.1.1Asegurar que los buckets S3 no sean públicos
Habilitar "Block Public Access" en la configuración de la cuenta y de cada bucket S3. Es uno de los errores más frecuentes de misconfiguration cloud.
AlmacenamientoNivel 1Crítico
2.1.2Cifrado de buckets S3 en reposo
Habilitar cifrado de servidor (SSE-S3 o SSE-KMS) en todos los buckets S3. Si se requiere gestión propia de claves, usar SSE-KMS con CMK.
AlmacenamientoNivel 2
3.1CloudTrail habilitado en todas las regiones
CloudTrail registra todas las llamadas a la API de AWS. Habilitar en todas las regiones, incluida la región global. Sin CloudTrail, no hay auditoría de lo que ocurre en la cuenta.
LoggingNivel 1Crítico
3.3CloudTrail integrado con CloudWatch Logs
Enviar los logs de CloudTrail a CloudWatch Logs para poder crear alertas en tiempo real sobre actividad sospechosa (creación de usuarios, cambios de IAM, etc.).
LoggingNivel 1
3.4Logs de CloudTrail cifrados con KMS
Cifrar los archivos de log de CloudTrail con una clave KMS de gestión propia para garantizar que solo el personal autorizado puede leerlos.
LoggingNivel 2
4.1Alertas sobre uso de la cuenta root
Crear una alarma CloudWatch que se active cuando se use la cuenta root. Cualquier uso de root es inusual y debe investigarse inmediatamente.
MonitoreoNivel 1
5.4Grupos de seguridad: no permitir 0.0.0.0/0 en SSH/RDP
No abrir los puertos 22 (SSH) o 3389 (RDP) a toda Internet. Usar bastión host, Systems Manager Session Manager o VPN para acceso administrativo.
RedesNivel 1Crítico
// CIS MICROSOFT AZURE FOUNDATIONS BENCHMARK v2.0
Organizado en secciones: Identity and Access Management, Microsoft Defender, Storage, Database, Logging, Networking, Virtual Machines, Key Vault, AppService.
1.1.1MFA requerido para todos los usuarios
Habilitar MFA mediante Conditional Access para todos los usuarios de Azure AD / Entra ID. Especialmente crítico para cuentas con roles de administrador global.
IAMNivel 1
1.1.4Sin Global Admins permanentes
Usar Privileged Identity Management (PIM) para activar roles de administrador bajo demanda (just-in-time), en lugar de asignaciones permanentes.
IAMNivel 2
2.1Microsoft Defender for Cloud habilitado
Habilitar Microsoft Defender for Cloud (antes Security Center) en todas las suscripciones para obtener el Secure Score y recomendaciones de seguridad.
DefenderNivel 1
3.1Cifrado en reposo para Storage Accounts
Azure cifra el almacenamiento en reposo por defecto (SSE). Usar Customer Managed Keys (CMK) en Azure Key Vault para mayor control criptográfico.
StorageNivel 1
3.7Acceso público a Storage Accounts deshabilitado
Deshabilitar el acceso de blob anónimo en todas las Storage Accounts. Controlar el acceso mediante Azure AD o SAS tokens con tiempo de expiración.
StorageNivel 1Crítico
5.1.1Diagnostic Settings habilitados
Habilitar Diagnostic Settings en todos los recursos Azure para enviar logs a Log Analytics Workspace, Event Hub o Storage Account para auditoría y SIEM.
LoggingNivel 1
5.2.1Alertas sobre operaciones administrativas
Crear Azure Monitor Alerts para actividades críticas: creación/eliminación de Key Vault, asignación de roles privilegiados, cambios en políticas de seguridad.
MonitoreoNivel 1
6.1RDP (3389) no expuesto desde Internet en NSG
Los Network Security Groups no deben permitir tráfico entrante en el puerto 3389 desde cualquier origen. Usar Azure Bastion o just-in-time VM access.
RedesNivel 1Crítico
8.1Rotación automática de claves en Key Vault
Habilitar la rotación automática de claves y secretos en Azure Key Vault. Definir política de expiración y alertas de vencimiento para todos los secretos.
Key VaultNivel 1
9.1HTTPS obligatorio en App Service
Forzar HTTPS en todas las aplicaciones de Azure App Service. Deshabilitar el tráfico HTTP y configurar TLS mínimo en 1.2 para todas las aplicaciones.
AppServiceNivel 1
// CIS GOOGLE CLOUD PLATFORM FOUNDATION BENCHMARK v2.0
Organizado en secciones: IAM, Logging, Networking, Virtual Machines, Storage, Cloud SQL, BigQuery, Kubernetes Engine.
1.1No usar Service Account Keys de largo plazo
Evitar las claves de Service Account de larga duración. Usar Workload Identity para Kubernetes, metadatos de instancia para Compute Engine. Eliminar claves no utilizadas.
IAMNivel 1
1.4MFA para todas las cuentas de Google Cloud
Requerir MFA para todas las cuentas de usuario que acceden a Google Cloud. Preferir MFA basado en hardware (FIDO2) para administradores.
IAMNivel 1
2.1Cloud Audit Logs habilitados
Habilitar Cloud Audit Logs (Admin Activity, Data Access, System Event) en todos los servicios del proyecto. Los logs de Admin Activity están activos por defecto; Data Access debe activarse explícitamente.
LoggingNivel 1
3.1Firewall: no permitir SSH/RDP desde 0.0.0.0/0
Las reglas de firewall no deben permitir tráfico entrante en los puertos 22 y 3389 desde cualquier origen. Usar Cloud IAP (Identity-Aware Proxy) para acceso administrativo seguro.
RedesNivel 1Crítico
5.1GCS Buckets no accesibles públicamente
Los buckets de Google Cloud Storage no deben ser accesibles públicamente sin necesidad explícita. Usar IAM conditions para control granular en lugar de allUsers/allAuthenticatedUsers.
StorageNivel 1Crítico
6.3.1Cloud SQL: no IPs públicas en bases de datos
Las instancias de Cloud SQL no deben tener IPs públicas asignadas. Acceder mediante Cloud SQL Auth Proxy o Private Service Connect desde la VPC interna.
Cloud SQLNivel 1
7.1GKE: no exponer el API server a Internet
Los clústeres de Google Kubernetes Engine deben tener el acceso al API server restringido a CIDRs específicos, o usar clústeres privados sin endpoint público.
GKENivel 1
7.3GKE: Workload Identity habilitado
Usar Workload Identity para que los pods de Kubernetes puedan acceder a APIs de Google Cloud sin usar Service Account Keys de larga duración en archivos.
GKENivel 1
// CONTROLES UNIVERSALES MULTI-CLOUD
Estos controles aplican independientemente del proveedor cloud. Son los primeros a implementar en cualquier entorno cloud antes de los controles específicos del proveedor.
MC-01Inventario completo de recursos cloud
Mantener un inventario actualizado de todos los recursos cloud (cuentas, suscripciones, proyectos, regiones). No puedes proteger lo que no sabes que existe. Usar CSPM para automatizar el descubrimiento.
VisibilityFundacional
MC-02Separación de cuentas por entorno
Entornos de desarrollo, staging y producción deben estar en cuentas/suscripciones/proyectos separados, no en el mismo con diferentes tags. El blast radius de un compromiso queda contenido.
GovernanceCrítico
MC-03Infrastructure as Code (IaC) con revisión de seguridad
Toda la infraestructura cloud debe definirse como código (Terraform, CloudFormation, Bicep). El código IaC debe escanearse con herramientas de seguridad (Checkov, tfsec) antes de aplicarse.
IaCDevSecOps
MC-04Secretos en un vault — nunca en variables de entorno
Claves API, contraseñas y certificados deben almacenarse en AWS Secrets Manager, Azure Key Vault o GCP Secret Manager. Nunca en variables de entorno, archivos de configuración o código fuente.
SecretsCrítico
MC-05Cifrado en reposo y en tránsito para todos los datos
Habilitar cifrado en reposo en todos los servicios de almacenamiento. Forzar TLS 1.2+ para todas las comunicaciones. Gestionar las claves de cifrado con KMS con rotación automática.
EncryptionFundacional
MC-06Logs centralizados y retenidos
Centralizar todos los logs cloud en un SIEM (o al menos en un bucket de log archiving separado y protegido contra eliminación). Retener mínimo 90 días online y 1 año en archivo.
LoggingFundacional

// Cloud Security Alliance — Cloud Controls Matrix v4

CSA Cloud Controls Matrix (CCM)

El CCM v4 de la CSA es el marco de controles de seguridad específicamente diseñado para entornos cloud. Define 197 controles organizados en 17 dominios. Es compatible con ISO 27001, SOC 2, PCI DSS y NIST CSF.

AIS Application & Interface Security 9 controles ▼
Seguridad en el diseño, desarrollo y despliegue de aplicaciones y APIs en la nube. Cubre desde los requisitos de seguridad hasta las pruebas de penetración de aplicaciones.
AIS-01
Políticas y procedimientos de seguridad de aplicaciones
Establecer políticas de desarrollo seguro de aplicaciones alineadas con estándares como OWASP, NIST y el ciclo de vida seguro de desarrollo (SSDLC).
AIS-03
Pruebas de seguridad de aplicaciones
Realizar evaluaciones de seguridad periódicas de las aplicaciones cloud, incluyendo análisis estático (SAST), dinámico (DAST) y pruebas de penetración.
AIS-05
Separación de entornos
Separar los entornos de desarrollo, prueba y producción. No usar datos reales de producción en entornos de desarrollo o prueba sin anonimizar.
AIS-07
Gestión de vulnerabilidades de aplicaciones
Mantener un proceso formal para identificar, evaluar, priorizar y remediar vulnerabilidades en aplicaciones cloud con SLAs definidos según severidad.
IAM Identity & Access Management 12 controles ▼
El dominio más crítico del CCM. La gestión de identidades y accesos es el primer control de seguridad en entornos cloud donde la identidad reemplaza al perímetro de red.
IAM-01
Políticas y procedimientos de IAM
Establecer políticas de gestión de identidades y accesos, incluyendo provisioning, modificación y baja de usuarios, con revisiones periódicas de accesos.
IAM-02
Autenticación fuerte
Implementar MFA para todos los usuarios, especialmente para el acceso a consolas de gestión cloud y funciones privilegiadas. Preferir FIDO2 sobre OTP por SMS.
IAM-05
Mínimo privilegio
Aplicar el principio de mínimo privilegio en todas las asignaciones de roles IAM. Usar permisos granulares en lugar de roles de administrador global. Revisar y revocar accesos no usados.
IAM-08
Gestión de credenciales de cuentas de servicio
Las cuentas de servicio no deben usar claves de largo plazo. Usar identidades gestionadas o roles temporales. Rotar las claves periódicamente y auditar su uso.
IAM-09
Segregación de funciones
Separar las funciones de gestión de la infraestructura cloud de las operaciones de negocio. Evitar que una sola persona pueda crear recursos y también aprobar los cambios.
IAM-12
Revisión periódica de accesos
Revisar y certificar los accesos IAM al menos trimestralmente. Revocar automáticamente accesos de usuarios que hayan dejado la organización o cambiado de rol.
DSP Data Security & Privacy 20 controles ▼
Protección de los datos en entornos cloud durante todo su ciclo de vida: creación, almacenamiento, transmisión, uso y eliminación. Incluye privacidad y cumplimiento regulatorio.
DSP-01
Clasificación de datos
Clasificar todos los datos según su sensibilidad (público, interno, confidencial, restringido) y aplicar controles de seguridad apropiados a cada nivel de clasificación.
DSP-04
Residencia y soberanía de datos
Conocer y controlar en qué regiones geográficas se almacenan los datos. Algunos datos regulados tienen restricciones de transferencia internacional que deben cumplirse.
DSP-07
Cifrado de datos en reposo
Cifrar todos los datos sensibles en reposo usando claves gestionadas por el cliente (CMK) en el KMS del proveedor cloud, con rotación automática de claves habilitada.
DSP-10
Cifrado de datos en tránsito
Todo el tráfico de datos debe cifrarse con TLS 1.2 o superior, incluyendo comunicaciones internas entre servicios cloud (cifrado este-oeste dentro de la VPC).
DSP-16
Eliminación segura de datos
Implementar procedimientos para eliminar datos de forma segura al finalizar su ciclo de vida, incluyendo la eliminación de copias en backups y logs que puedan contener datos sensibles.
IVS Infrastructure & Virtualization Security 11 controles ▼
Seguridad de la infraestructura cloud: redes virtuales, sistemas operativos, contenedores y hipervisores. Cubre segmentación de red y hardening de la capa de infraestructura.
IVS-01
Segmentación de red
Implementar segmentación de red virtual (VPCs, subnets, security groups) para separar entornos y tipos de carga de trabajo. Controlar el tráfico entre segmentos con reglas de firewall.
IVS-04
Hardening de instancias
Aplicar configuraciones de hardening (CIS Benchmarks) a todas las instancias cloud. Eliminar servicios, puertos y usuarios innecesarios de las imágenes base.
IVS-07
Seguridad de contenedores
Escanear imágenes de contenedores en busca de vulnerabilidades y configuraciones inseguras. No ejecutar contenedores como root. Usar políticas de red en Kubernetes.
IVS-09
Gestión de capacidad y disponibilidad
Monitorizar el uso de recursos y planificar la capacidad para garantizar disponibilidad. Implementar auto-scaling y arquitecturas multi-region para alta disponibilidad.
LOG Logging & Monitoring 13 controles ▼
Registro de actividad, monitoreo de seguridad y gestión de eventos en entornos cloud. Sin visibilidad completa, la detección y respuesta a incidentes es imposible.
LOG-01
Registro de actividad de APIs cloud
Habilitar y retener los logs de actividad de API en todos los servicios cloud (CloudTrail, Activity Log, Cloud Audit Log). Incluir logs de acceso a datos sensibles.
LOG-05
Monitoreo de seguridad
Implementar monitoreo continuo de seguridad con alertas automáticas para eventos críticos. Definir y documentar las reglas de detección y los umbrales de alerta.
LOG-09
Protección de integridad de logs
Proteger los logs contra modificación o eliminación. Los logs deben enviarse a un destino inmutable (bucket con Object Lock) separado de los sistemas que generan los logs.
LOG-11
Retención de logs
Definir y aplicar políticas de retención de logs que cumplan los requisitos regulatorios y de respuesta a incidentes. Mínimo 90 días online; 1 año en archivo para cumplimiento.

Checklist de Seguridad Cloud

Controles fundamentales a implementar inmediatamente en cualquier entorno cloud, ordenados por prioridad. Los primeros 6 reducen el 80% del riesgo con el menor esfuerzo.

✓
MFA EN TODAS LAS CUENTASHabilitar MFA para todos los usuarios, empezando por las cuentas root/globales y continuando con todos los usuarios con acceso a consola.
✓
ACTIVAR LOGS Y AUDIT TRAILSHabilitar CloudTrail / Activity Log / Cloud Audit Log en todos los servicios y regiones. Enviar a destino inmutable separado.
✓
ELIMINAR ACCESO PÚBLICO ACCIDENTALRevisar y corregir buckets de almacenamiento públicos, instancias con IPs públicas innecesarias y puertos SSH/RDP abiertos a 0.0.0.0/0.
✓
PRINCIPIO DE MÍNIMO PRIVILEGIO IAMRevisar todos los roles y políticas IAM. Eliminar permisos wildcard (*) y roles de administrador global no justificados.
✓
CIFRADO EN REPOSO Y TRÁNSITOHabilitar cifrado en todos los servicios de almacenamiento. Usar TLS 1.2+ en todas las comunicaciones. Rotar claves periódicamente.
✓
SECRETOS EN VAULT — NO EN CÓDIGOMover todas las credenciales hardcodeadas a AWS Secrets Manager / Azure Key Vault / GCP Secret Manager. Escanear repos con TruffleHog.
✓
CSPM — POSTURE MANAGEMENT ACTIVOActivar Security Hub / Defender for Cloud / Security Command Center para visibilidad continua del estado de seguridad y detección de misconfiguraciones.
✓
ALERTAS SOBRE ACTIVIDAD CRÍTICACrear alertas para: uso de cuenta root, cambios de IAM, creación de recursos en regiones no usadas, cambios de configuración de seguridad.
✓
SEPARAR CUENTAS POR ENTORNOProducción, staging y desarrollo en cuentas/suscripciones separadas. Aplicar Service Control Policies u equivalente para governance centralizado.
✓
IaC CON ESCANEO DE SEGURIDADTodo el código de infraestructura escaneado con Checkov o tfsec antes de aplicarse. Sin recursos cloud creados manualmente en producción.
✓
INVENTARIO DE ACTIVOS CLOUD ACTUALIZADOCMDB o CSPM con inventario en tiempo real de todos los recursos cloud en todas las cuentas y regiones. Base del modelo de responsabilidad compartida.
✓
PLAN DE RESPUESTA A INCIDENTES CLOUDPlaybooks específicos para: cuenta comprometida, bucket público, instancia con malware, exfiltración de datos. Integrar con Ley 21.663 (3h/72h/15d).

Relación con Otros Marcos

La seguridad cloud no opera de forma aislada. El CSA CCM mapea directamente con los estándares más importantes, facilitando el cumplimiento simultáneo de múltiples requisitos.

ISO/IEC 27001:2022
SGSI en la Nube
Los controles del Anexo A de ISO 27001 mapean directamente al CCM de la CSA. El control 5.23 (SI para uso de servicios cloud) de ISO 27001:2022 es el nexo central. El CCM proporciona el detalle técnico operativo que ISO 27001 no especifica para entornos cloud.
NIST CSF 2.0
Función GOVERN y PROTECT
Las subcategorías GV.SC (riesgo de cadena de suministro cloud), PR.IR-01 (redes protegidas) y PR.DS (seguridad de datos) del CSF se implementan con los controles del CCM y los CIS Benchmarks de cada proveedor.
Zero Trust (NIST SP 800-207)
Identidad como Perímetro
Los pilares de Identity y Applications de Zero Trust se implementan directamente en la nube con los controles IAM del CCM y los CIS Benchmarks. Cloud es el entorno natural para Zero Trust — todo acceso ya es "remoto".
Ley 21.663 Chile
OIV con Infraestructura Cloud
Los OIV que operan infraestructura crítica en la nube deben aplicar los controles del CCM y los CIS Benchmarks como parte de su SGSI. El control SEF del CCM se alinea directamente con los plazos de reporte de la Ley 21.663.
NIST SP 800-86 (Forense)
Forense Digital Cloud
El control SEF-05 del CCM y el NISTIR 8006 (Cloud Forensics) extienden el SP 800-86 a entornos cloud. La evidencia cloud (logs de CloudTrail, snapshots) tiene características únicas de recolección y preservación.
SOC 2 Type II
Trust Service Criteria
Los clientes de servicios cloud SaaS frecuentemente requieren a sus proveedores certificación SOC 2 Type II. El CCM de la CSA mapea directamente a los Trust Service Criteria (Security, Availability, Confidentiality) facilitando la preparación para la auditoría.
